Adopting an existing project

So far you've started fresh — an idea, then a design, then a build. But what if you already have a pile of code you made by prompting and hoping? There's one command for exactly that situation, and this lesson is about when to reach for it.

What adopt is for

Most of the verbs you've met assume you're starting from nothing. /hacky-hours adopt is different. It is, in the framework's own words, "the only operation on existing code" in version 4. Everything else builds something new; adopt meets something that's already there.

A quick definition before we go on. To adopt a codebase means to bring an existing project into the Hacky Hours framework — so the team of agents can read what you've built, document it, and help you take it forward with the same rigor a new project gets.

There is no "productionize" verb You might expect a single button that makes code production-ready. There isn't one — and that's honest. The framework's view is that production quality "emerges from continuous ensemble play after adoption." In plain terms: you adopt first, then keep working with the team. The quality grows from the ongoing work, not from one command.

What it actually does

When you run /hacky-hours adopt inside a folder that has code, the whole team "meets" your codebase. Here's the shape of it, without the technical detail:

  1. It asks four quick questions — what is this, how serious is it (a rigor "tier" from a weekend tool up to regulated software), what voice you want, and any files the team shouldn't read.
  2. Each agent reads the code through its own lens — Security looks for risky patterns, Accessibility checks the interface, and so on. Each one reports back how involved it thinks it should be and what it found.
  3. You review the consolidated assessment — a single report with everyone's findings and a punch list sorted by priority. You confirm or adjust before anything is written.
  4. The framework writes baseline documents — the same kind of design docs a new project would have, plus a record of what the team found.

What adoption gives you — and what it doesn't Adoption produces a thoughtful first read of your code and a prioritized list of things to look at. It does not guarantee your code is correct, safe, or finished. The findings are starting points written by AI agents — useful direction, not a clean bill of health. You still own the decision of what to fix and the responsibility for what you ship.

The "vibe-coded" project

A lot of people arrive here with what's often called a "vibe-coded" project — something built by prompting an AI, accepting whatever it produced, and never writing down a plan. It might even work. But there's no design doc, no record of decisions, and no clear sense of what's risky.

That's the ideal case for adopt. It takes the thing you already have and brings it up to the standard you'd get if you'd done the full flow from the start — without making you throw it away and begin again.

Adopt is how a "it just sort of happened" project becomes one you can actually reason about.

Going deeper 🌱

Adoption produces design docs and, when something is decided, Architecture Decision Records — short notes that capture why a choice was made. They're worth learning to read.

Good next steps
  • Learn what an ADR is and why teams keep them: adr.github.io. You'll see them generated throughout this module.
  • Run /hacky-hours adopt on any old project you have lying around — even a finished one — and read the assessment. It's the fastest way to feel what the team sees.
  • Browse the framework docs: github.com/empathetech/hacky-hours-docs.

What you just learned

  • /hacky-hours adopt is the only operation that works on existing code.
  • The team reads your code, reports findings, and writes baseline design docs after you confirm.
  • There's no single "productionize" button — quality comes from continuing to work with the team.
  • Adopt is perfect for a "vibe-coded" project you want to bring up to standard.
  • If there's no code yet, start at Step 1 (ideate) instead.
