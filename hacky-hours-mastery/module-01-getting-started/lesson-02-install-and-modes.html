The honest note on install scripts Piping a script straight into bash or iex runs whatever that script contains. That's standard practice for many tools, but the responsibility to trust the source is yours. If you'd rather read it first, open the install.sh or install.ps1 file on GitHub before running it. Hacky Hours can't make that judgment call for you.

Builder mode vs. engineer mode

This is the setting that matters most for how Hacky Hours feels to you. It has two modes — two ways of talking — and you choose which one fits you:

  • Builder mode — plain language. It explains tradeoffs through outcomes and real-world analogies, and never uses jargon without defining it first. This is the default for every new project, and it's where most non-engineers will want to stay. As the skill puts it, builder mode "is not less rigorous — it's the same information grounded in what the user needs to decide."
  • Engineer mode — technical language. It assumes you're familiar with programming concepts, frameworks, and tooling, and talks in precise terms — things like type safety, dependency footprint, and architectural patterns. It does less hand-holding because it assumes you don't need it.

The question: "What database should I use for my app?"

Not sure which to pick? Start in builder mode. It's the default for a reason, and you lose nothing — the rigor is the same. If it ever feels like it's over-explaining things you already know, flip to engineer mode in one command.

One more thing: how it gets started (Mode A vs. Mode B)

"Mode" is doing double duty in Hacky Hours, so let's clear up the second meaning, because it'll save you confusion. Separate from builder/engineer, the skill behaves differently depending on how it was started:

  • Mode A — you invoked it. You typed /hacky-hours (with or without a command after it). You opted in, so it proceeds straight to work.
  • Mode B — context triggered it. You wrote something in plain English like "build me an app" and Claude recognized a build-a-real-thing moment. Because you didn't type the command yourself, it stops and shows you an invitation first — offering to engage the full team, sketch informally, or skip the framework — and waits for you to choose. It won't start scaffolding or asking project questions until you say so.

The takeaway: Hacky Hours never barges in. If it appears uninvited, it'll ask permission before doing anything real.

Going deeper 🌱

The idea that a tool should meet you at your level — explaining more or less depending on what you know — is a thread that runs through good documentation everywhere.

A door worth walking through If the builder/engineer split interests you, read the Diátaxis framework — a widely used, free guide to writing technical documentation that adapts to whether the reader is learning, doing, or looking something up. It's the same instinct as Hacky Hours' modes, applied to writing.

What you just learned

  • Install with one line — curl … install.sh | bash on macOS/Linux, irm … install.ps1 | iex on Windows — then restart Claude Code.
  • Run /hacky-hours help to see every command.
  • Builder mode is plain-language (the default); engineer mode is technical and assumes more. Switch with /hacky-hours tools mode.
  • Builder mode isn't less rigorous — just framed around what you need to decide.
  • Mode A (you typed the command) proceeds; Mode B (context-triggered) shows an invitation and waits for your choice.
