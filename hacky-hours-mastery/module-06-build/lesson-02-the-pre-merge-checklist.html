The pre-merge checklist

Before any change becomes part of your real app, Step 4 runs it past five questions. They're short, they're plain, and they catch the mistakes that hurt beginners most. Let's go through all five — and be honest about what passing them does and doesn't mean.

8 min read hands-on grows: review habits
Read this first — what passing the checklist means Running this checklist helps you avoid some common, well-known mistakes. It is not proof that your app is safe, private, legal, or bug-free. No checklist can promise that. If you're handling sensitive data — health, money, anything that could hurt someone if leaked — get a qualified human review as well. Passing this list is a floor, not a finish line.

The five items, verbatim

The Step 4 pre-merge checklist
  • Implementation matches the relevant design document
  • No secrets, API keys, or credentials in code or commit history
  • All user input that crosses a trust boundary is validated
  • Error messages don't expose internal system state
  • Change has been manually tested against the relevant user journey

Each item, in plain language

1. Implementation matches the relevant design document

Did the team build the thing you actually designed in Step 2 — not a different thing it decided was easier? You can check this without reading code: ask it to describe what the change does and compare that to your design doc.

2. No secrets, API keys, or credentials in code or commit history

A secret is any value that proves identity to another service — a password, an API key, a token. None of these should ever be written into the code or its saved history, because once committed, they can leak publicly. (We covered this with the security agent in Module 4.)

3. All user input that crosses a trust boundary is validated

A trust boundary is the line where data from the outside world (a form, a URL, an upload) enters your app. "Validated" means the app checks that input is the right shape and safe before using it — instead of trusting whatever a stranger typed.

4. Error messages don't expose internal system state

When something breaks, the message a user sees should be friendly and vague ("Something went wrong"), not a wall of internal detail that hands an attacker a map of how your app works.

5. Change has been manually tested against the relevant user journey

Someone actually clicked through the real steps a person would take and confirmed it works. Not "the code looks right" — actually used it. For a non-engineer, this is often your strongest contribution, and the next lesson is all about it.

The stay-in-charge habit

You don't have to drive every step by hand — but you do stay in charge of the standard. The habit is one plain sentence, said before any merge:

"Run the pre-merge checklist before we merge — and tell me anything you're unsure about."

That second half matters as much as the first. Asking the AI to name its uncertainty turns a silent "all good" into a real conversation — and the things it flags are exactly where your attention belongs.

A ticked box is a claim, not a guarantee Ticking every item here means "I checked these five things." It does not certify your app is secure or correct. Treat a fully-ticked checklist as permission to merge a small change with reasonable confidence — never as a substitute for real review on anything sensitive.

Growing your review habits 🌱

Good next steps
  • Skim the OWASP Top 10 — you'll see items 2, 3, and 4 of this checklist appear there as the most common web risks.
  • For the next change you merge, write down which checklist item you understood least, then ask your team to teach you that one.
  • Run /hacky-hours audit on a project — it re-checks several of these items across your whole codebase, not just one change.

What you just learned

  • Step 4 runs five checks before any change merges — you saw all five verbatim.
  • In plain terms: it matches the design, hides secrets, validates input, stays quiet in errors, and was actually tested by a human.
  • The habit: ask the AI to run the checklist and to say what it's unsure about.
  • Passing the checklist avoids common mistakes — it does not prove your app is safe.
  • For sensitive data, add a qualified human review on top.
