Working with the security agent

You don't have to be a security expert to reduce common risks in what you build. But you do have to stay in the loop — because the responsibility for what your software exposes is, and always will be, yours. Let's get clear on both halves of that.

What the security agent is for

When you run /hacky-hours, one of the twelve agents on the team is Alex Davies — Security 🛡️. Alex's job is to raise the uncomfortable questions you might not think to ask: What happens if someone types something malicious into this box? Where are we keeping the password? What can a logged-in stranger reach that they shouldn't?

One of the framework's eight guiding values sets the tone here:

From the Hacky Hours values "Reduce attack surface. When two approaches accomplish the same goal, prefer fewer credentials, fewer external dependencies, and fewer moving parts."

That's a useful instinct — often the most secure choice is simply the simpler one. But an instinct isn't a security review. The agent helps you avoid obvious mistakes and prompts you to think; it can't promise it caught everything, and it doesn't know your specific situation the way you do.

A common, costly mistake: secrets in your code

A secret is any value that proves who you are to another service: an API key, a database password, a token. A frequent and damaging mistake is writing a secret directly into your code — and then pushing it to GitHub, where others can read it.

Automated bots scan public code constantly. A leaked cloud key can be found and abused within minutes, sometimes running up large charges before anyone notices.

Avoid this Anything secret belongs in an environment variable — a value stored outside your code, in a .env file that git is told to ignore. The code reads the value at runtime; the secret itself never enters your repository.

Where security shows up in a build

Security in Hacky Hours isn't a single phase you bolt on at the end. It comes up at three natural moments — knowing them helps you stay involved instead of hoping it was handled:

  1. In Design (Step 2) — the agent helps you write a SECURITY_PRIVACY.md: what data you collect, who can do what, and a baseline of checks.
  2. In Build (Step 4) — every change runs through a pre-merge checklist before it's called done.
  3. In Audit — /hacky-hours audit scans your project for leaked secrets and other common red flags.

Here's the actual pre-merge checklist, straight from the skill:

The Step 4 pre-merge checklist
  • Implementation matches the relevant design document
  • No secrets, API keys, or credentials in code or commit history
  • All user input that crosses a trust boundary is validated
  • Error messages don't expose internal system state
  • Change has been manually tested against the relevant user journey

You don't memorize it — you just make sure it actually runs. When a build feels "done," a good habit is one plain sentence:

"Run the pre-merge checklist before we merge — and walk me through anything you're unsure about."

The honest limit Passing this checklist means you avoided some well-known mistakes. It does not mean your app is safe to handle sensitive data, comply with privacy laws, or withstand a determined attacker. If you're building something where a breach would genuinely hurt people, get a real security review. Knowing where your own knowledge ends is itself a security skill.

Learning more on your own 🌱

This is where you go from using a tool to understanding the craft. None of this is required to finish the course — but every link is a door, and walking through a few is how you grow into someone who can make these calls with real judgment.

Good next steps
  • Read the OWASP Top 10 — the industry's plain-language list of the most common web vulnerabilities. You'll recognize the agent's questions in it.
  • Run /hacky-hours audit on a project you've started and read every flag, including the ones you don't fully understand yet — then look them up.
  • Read how the agent thinks: /hacky-hours team show alex --prompt.

What you just learned

  • The security agent helps you ask better questions — it doesn't guarantee safety, and the responsibility stays yours.
  • Secrets never live in code; they live in environment variables.
  • You can now recognize a leaked secret on sight.
  • Security shows up in Design, Build (the pre-merge checklist), and Audit.
  • For anything genuinely sensitive, a real security review is still the right call.
