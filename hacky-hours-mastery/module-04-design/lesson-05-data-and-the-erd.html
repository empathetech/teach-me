The data agent & the ERD

Almost every product remembers something — a user, an order, an event. This lesson introduces the agent who decides what gets stored and how it connects, explains what a data model and an ERD actually are, and is clear about whose responsibility the stored data is.

9 min read hands-on grows: data literacy
Read this first — data about people is your responsibility The data agent helps you store information sensibly and collect less of it. It does not make you compliant with privacy laws, and neither does this lesson. The moment your product stores information about real people, the responsibility for what you keep, how you protect it, and what you're allowed to do with it is yours. This lesson helps you make better decisions — not skip the ones that matter.

Databases, tables, and relationships

A database is an organized place a product stores its information so it can find it again later. Most databases are made of tables — think of each table as a single spreadsheet for one kind of thing.

Each kind of thing your product remembers is an entity. A User is an entity; an Event is an entity. Each entity becomes a table. The columns of that table are its fields — the specific pieces of information you keep, like a user's email or an event's date.

The interesting part is the relationships: how entities connect. One user might create many events. One event might have many attendees. Mapping those connections is most of what a data model is.

What an ERD is

An ERD — entity-relationship diagram — is a picture of your entities and the relationships between them. Boxes are entities; lines between them show how they connect. It's the fastest way to see the shape of your data at a glance.

Yuki generates the ERD as a Mermaid diagram — that plain-text diagram format GitHub renders into a picture — so it lives in your project's files and updates as text. Here's a tiny one in words: a User box, an Event box, and a line between them reading "one user creates many events."

The value: privacy-preserving before data-rich

This is where Yuki's strongest instinct comes in, straight from the framework's values:

From the Hacky Hours values "Privacy-preserving before data-rich. Collect the minimum data needed. Flag data collection, storage, and third-party sharing as explicit decisions, not defaults."

In plain terms: don't collect information just because you can. Every field you store about a person is something you then have to protect, justify, and eventually delete. The safest data is the data you never collected. Yuki will push you to ask, of each field, do we actually need this?

The minimum data that does the job is almost always the right amount.

Learning more on your own 🌱

Understanding data is one of the most transferable skills you can build — it shows up in spreadsheets, products, and almost every software job.

Good next steps
  • Read freeCodeCamp's DBMS and SQL basics — a friendly intro to databases and how you ask them questions.
  • On a real idea, list every field you'd want to store, then cross out the ones you don't actually need yet. That's the privacy-preserving instinct in action.
  • Read how the agent thinks: /hacky-hours team show yuki --prompt.
