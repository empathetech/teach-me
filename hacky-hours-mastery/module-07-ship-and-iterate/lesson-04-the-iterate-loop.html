The iterate loop (Step 5)

You shipped. Real people are using the thing you built. Now what? You iterate — listen, decide what matters, update your docs, and build the next round. This is Step 5, and it's the loop that turns a one-time launch into a living product.

Jordan, one last time 🚀. To iterate means to improve something through repeated rounds — ship, learn, adjust, ship again. Step 5 gives that loop a shape so it doesn't become a pile of half-remembered "I should fix that someday" notes.

Why a step, and not just more chatting?

This is the question worth sitting with: you're already in a Claude Code session with the whole team right there. Why not just say "hey, also fix this" and keep going? Why bother with a named step, an ITERATION.md, and updating documents?

Because of a quiet failure mode called drift.

A real example Someone was troubleshooting a small thing — back and forth with Claude, a few times, just chatting. Each little fix seemed fine on its own. But the patches slowly pulled the code away from what the design docs actually said, and nobody noticed until he looked up and realized the app no longer matched the plan. What finally caught it wasn't more chatting — it was pointing out that the work had drifted from Hacky Hours, and re-anchoring to the design.

That's drift in one sentence: a series of small, reasonable, ad-hoc changes that each make sense alone but add up to a project that no longer matches its own plan. It's the default outcome of pure back-and-forth chat, because a chat optimizes for the last thing you said — not for the project as a whole.

Ad-hoc chatting optimizes for the next message. A defined step optimizes for the project.

Here's the difference, side by side:

Ad-hoc back-and-forth Running /hacky-hours iterate
The design docs say one thing; the code quietly becomes another. No record of why. You amend the docs first, so the plan and the code stay in sync — on purpose.
Fixes are scattered across a chat you'll never re-read. Everything lands in ITERATION.md, then gets triaged with intention.
"Why is it like this?" — nobody knows six weeks later. Significant decisions get an ADR — a written "why" you can return to.
The team's checks (security, accessibility) get skipped because you never re-entered the flow. You re-enter the Build cycle, so the pre-merge checklist and the team's eyes apply again.
The deeper point: your design docs are the project's source of truth. Ad-hoc chat changes the code and leaves the truth behind; the iterate step changes the truth first, then the code follows. That single ordering is what keeps a project honest over months. (It's also why the pre-merge checklist's very first item is "implementation matches the relevant design document.")
You're still allowed to just chat None of this means quick conversation is forbidden — exploring and asking questions in the session is great. The line to watch is when chatting starts changing what the product is without anything writing that down. When a change is real enough to matter, that's the moment to run iterate instead of firing one more message. Knowing which is which is a judgment you'll grow.

The five moves of Step 5

When you run /hacky-hours iterate (or step 5), the Ops agent walks you through five moves in order:

  1. Capture. Brain-dump everything — bugs, complaints, feature requests, things you'd change now that you've seen it live. No filtering. It all goes into a file called ITERATION.md. The agent first checks your feedback/ folder and summarizes anything users already left.
  2. Synthesize. The agent reads your brain-dump against your existing design docs and asks, for each item: does this change how the product works (amend a design doc)? introduce a new decision (note an ADR)? or is it just an implementation fix (straight to the backlog)?
  3. Prioritize. Sort each item into one of three buckets — we'll practice this below.
  4. Amend. Update the design docs that need it, and write ADRs for the significant decisions.
  5. Build. Head back into the Step 4 build cycle with your updated backlog.

The three prioritize buckets are worth memorizing, because you'll use them constantly:

The three triage buckets — straight from the skill
  • Hotfix — "broken in production, needs immediate attention."
  • Next milestone — "important enough to be in the next planned release."
  • Backlog — "valid but not urgent; add to BACKLOG.md without a milestone."

What an ADR is, and when to write one

An ADR is an Architecture Decision Record — a short note that captures a meaningful decision and why you made it. Not every change needs one. You write an ADR when a decision is significant: when a future you (or a teammate) would otherwise look at the project and wonder "wait, why did we do it this way?"

Capturing the "why" at the moment you decide is one of the kindest things you can do for your future self. The reasons are obvious now; they won't be in six months.

Iterate vs pivot

Most of the time you iterate: same product, made better. Occasionally something bigger happens, and the right move is a pivot — a deliberate change of direction (handled by the legacy review 3).

The line is clear. You pivot, rather than iterate, when the core changed:

  • Audience — you built it for hobbyists, but small businesses are the ones who actually need it.
  • Problem — the thing you set out to solve turned out not to be the real pain; a different one is.
  • Form-factor — you built a website, but it really wants to be a mobile app or a command-line tool.

If the core audience, problem, or form-factor changed, that's a pivot. Otherwise, you're iterating.

Iterating doesn't reset your responsibilities Every iteration cycle ships new code to a live, public app. Run /hacky-hours audit again before each release — a fix can introduce a fresh leaked secret or expose new data just as easily as the first launch did. "We already shipped once safely" is not the same as "this change is safe."

Closing the loop 🎉

That's the whole arc: ideate → design → roadmap → build → iterate → build → iterate …. You don't graduate out of the loop — you live in it, and each lap your product gets a little more solid. When you finish a Step 5 cycle, the agent archives your ITERATION.md and your backlog reflects reality again.

Try it in your terminal After your next release, sit down and run: Let it pull your feedback, brain-dump freely, and triage together. That single habit — capture, decide, amend, build — is what separates a project that fizzles from one that keeps getting better.

You've now walked the entire build-and-ship arc, from a blank page to a live product you can keep improving. That's worth celebrating.

What you just learned

  • A defined step beats ad-hoc chatting because it prevents drift — small fixes silently pulling the code away from the plan.
  • Step 5 is five moves: Capture → Synthesize → Prioritize → Amend → Build, with everything captured in ITERATION.md.
  • Triage into three buckets: hotfix, next milestone, backlog.
  • Write an ADR (Architecture Decision Record) to capture the "why" behind significant decisions.
  • You pivot (not iterate) when the core audience, problem, or form-factor changed.
  • Run an audit before every release — iterating doesn't reset your responsibilities.
