The three-lane audit

Before you ship, it helps to stop and ask: is this actually ready? Hacky Hours has a single command for that — /hacky-hours audit. It looks, it reports, and — importantly — it never touches your files. Let's learn to read what it tells you.

What the audit does

Run /hacky-hours audit and the Ops agent performs a read-only scan of your project, then gives you back a single prioritized report. The key word is read-only:

Straight from the skill The audit is "a read-only scan that tells the user exactly what needs to happen before they can ship. Never modifies any files except the optional scorecard. Safe to run at any time."

It's a three-lane audit — code + docs + cross-reference — with a traffic-light scorecard.

Because it never changes anything, you can run it as often as you like, whenever you're unsure. It's the natural thing to do right before a release.

The three lanes

"Three-lane" means the audit looks at your project from three angles at once:

  1. Code — the highest-stakes check first: are there secrets (API keys, passwords, tokens) sitting in your code or git history? Is .env properly ignored? What's uncommitted or unpushed?
  2. Docs — are your design documents actually filled in? Each gets a status: filled in, placeholder only, or missing. Is there a CHANGELOG entry for the version you're about to release? Is there a LICENSE?
  3. Cross-reference — do the pieces agree with each other? For example, does the version named in the skill match the latest CHANGELOG version? Do your GitHub Issues line up with your BACKLOG.md?

One run, three angles, one ordered to-do list at the end: blockers first, warnings next, suggestions last.

The traffic-light scorecard

The audit reports each finding with a color you already understand from the road. A tag here is just a short label git can attach to a moment in your project — you'll meet those next lesson; for now, focus on the colors:

  • 🟢 Green — looks good. No obvious problem found in this lane.
  • 🟡 Amber — a warning. Not a dealbreaker, but worth handling before you ship.
  • 🔴 Red — a blocker. Stop and fix this before going further, especially anything about leaked secrets.

A scorecard doesn't make decisions for you. It points; you choose what to do.

Green is not a guarantee The skill itself says the secrets scan is "a heuristic check, not a security guarantee." Heuristic means it checks for common, known patterns — it can miss things it wasn't designed to look for. An all-green scorecard means the automated checks found no obvious problems. It does not mean your app is secure, private, or lawful to operate. For anything that handles real people's data, a green light is a starting point, not a sign-off.

A note on review 1, 2, 3

If you see older Hacky Hours material mention review 1, review 2, and review 3, here's the map. These are the legacy commands for working with existing or inherited code:

  • review 1 (audit) — the older name for what /hacky-hours audit now does. Prefer audit going forward.
  • review 2 (optimize) — clean up code or docs that have grown oversized or stale.
  • review 3 (pivot) — rework a project whose direction has changed (we'll come back to pivoting in the last lesson).

Going further 🌱

The best way to learn the scorecard is to run it on something real and read every single finding — including the ones you don't fully understand yet, then look those up.

Try it in your terminal Open a project you've started with Hacky Hours and run: Read the report top to bottom. When it offers to save a scorecard to audits/, say yes — it's a dated record of where the project stood, and a good habit before any release.

What you just learned

  • /hacky-hours audit is a read-only scan — it reports, it never modifies your files.
  • Three lanes: code (secrets, git state), docs (design docs, CHANGELOG, LICENSE), cross-reference (versions and Issues agree).
  • Findings come back as a 🟢/🟡/🔴 traffic-light scorecard, ordered blockers-first.
  • Green means the automated checks passed — not that your app is secure.
  • The legacy review 1/2/3 commands map to audit / optimize / pivot.
