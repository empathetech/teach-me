Semantic versioning: patch, minor, major

Version numbers aren't random. The common scheme is semantic versioning, written as three numbers: MAJOR.MINOR.PATCH — for example 1.4.2. Each number means something specific, and "bumping" a version means increasing one of them:

  • Patch (1.4.2 → 1.4.3) — you fixed a bug or made a small correction. Nothing new, nothing broken.
  • Minor (1.4.2 → 1.5.0) — you added a new feature, but existing things still work the same way. The patch number resets to 0.
  • Major (1.4.2 → 2.0.0) — you made a breaking change: something that worked before now works differently or is gone. Both lower numbers reset.

Two milestones have conventional numbers worth remembering:

The two numbers you'll meet first
  • MVP — 0.1.0. A version starting with 0. signals "early, things may still change." Your first shippable cut usually lands here.
  • V1 — 1.0.0. Reaching 1.0.0 is a statement: "this is stable enough to depend on."

What update 1 actually does

When you run /hacky-hours update 1, the Ops agent publishes a release for you, step by careful step. It writes to a shared, public system, so the skill is explicit: "Always confirm before taking any action. Never run any action without explicit confirmation." Here's the sequence:

  1. Check the gh CLI is ready. The gh CLI is GitHub's official command-line tool. The agent runs gh auth status; if you're not logged in, it stops and tells you to run gh auth login first.
  2. Read the latest CHANGELOG entry. It pulls the most recent version block from your CHANGELOG.md and shows it: "Is this the release you want to publish?"
  3. Confirm the tag. A tag is a label git attaches to one exact commit — a permanent bookmark for "this is where version 1.5.0 lives." The agent shows you the exact git tag and git push commands and asks before running them.
  4. Preview. It shows precisely what will be published and reminds you this becomes visible on your repo's Releases page.
  5. Publish — only after you confirm. It runs gh release create with your version and CHANGELOG notes, then tells you where to find the result.
Try it in your terminal Once your CHANGELOG has an entry and your audit is clean: It will pause at each step and wait for your go-ahead. Read every preview before you confirm — nothing is published until you say so.
This goes out in public — and stays there A release is published to your repository for the world to see, and a git tag is meant to be permanent. Two things follow:
  • Read every preview the agent shows you. Confirm only what you actually intend to publish. The framework will never publish without your explicit yes.
  • If a release accidentally contains a secret, deleting the release is not enough — treat the secret as compromised and rotate it. Run an audit before you cut the release, not after.

update 2: syncing your backlog

A close cousin is /hacky-hours update 2. Your BACKLOG.md is your local list of planned work; GitHub Issues are the same kind of list, but living online where collaborators can see them. Over time the two drift apart.

update 2 does a two-way sync between them. Neither one is treated as the boss — the skill says "the user resolves all conflicts at sync time." It sorts everything into buckets (new in the backlog, new in Issues, changed in both, or done) and asks you to confirm before it creates, updates, or closes anything.

Publishing and planning both follow the same rule: the framework proposes, you decide, nothing happens without your yes.

Going further 🌱

Learn the versioning standard Semantic versioning is a real, written specification — short and readable. If you want the exact rules behind patch / minor / major, read it at semver.org. You'll find the same MAJOR.MINOR.PATCH meanings you just practiced.
