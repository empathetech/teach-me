Deployment & the cloud

You've built something on your own computer. Now you want other people to be able to use it. That step has a name — deployment — and it's less mysterious than it sounds. Let's walk through what actually happens, and what changes the moment you go live.

What "deploy" actually means

To deploy is to make your app available on the internet so that real people, on their own devices, can open it. Up to now your project has run on your laptop — only you can see it. Deploying copies it onto a computer that's always on and connected to the internet, and gives it a web address (a URL) anyone can visit.

That's the whole idea. The word sounds heavy, but it's just "put it where others can reach it."

"The cloud" in plain terms

People say software runs "in the cloud" as if it floats somewhere abstract. It doesn't. The cloud is simply someone else's computers, in a building somewhere, that you rent a slice of. Instead of buying and maintaining a server yourself, a company runs the machines and lets your app live on them. You pay for what you use — often nothing at all for small projects.

Why this fits the Hacky Hours values One of the framework's guiding principles is "less infrastructure before more — prefer a simple hosted solution over a self-managed one." Renting a slice of someone's well-run machines is almost always simpler and safer than operating your own server. Let someone whose full-time job is keeping computers online do that part.

Hosting: push to GitHub, it goes live

Hosting is the service that holds your deployed app and serves it to visitors. For the kind of web projects you'll build early on, two popular, free-to-start hosts are Netlify and Vercel. They work in a way that fits beautifully with everything you've already learned about git and GitHub.

You connect your GitHub repository (your project's online home) to the host once. After that, the flow is automatic:

  1. You write or change code on your computer.
  2. You git push to GitHub, exactly as you already do.
  3. The host notices the new code and builds it — assembles the final files visitors will receive.
  4. Within a minute or two, the live URL updates. Your change is online.

You don't "upload a website." You push code, and the host turns it into a live site for you.

That automatic step — code change to live site with no manual upload — is sometimes called continuous deployment. You don't need the term. You just need to know that once it's wired up, shipping is one git push.

Secrets in production: environment variables

Back in Module 4, Alex (Security) showed you that secrets — API keys, passwords, tokens — never belong in your code. That rule doesn't relax when you deploy; it matters more, because your code is now reachable from the whole internet.

An environment variable is a named value your app reads at runtime, stored outside the code. On your laptop it lives in a .env file that git ignores. In production, it lives in your host's settings — Netlify and Vercel both have a dashboard page literally called "Environment variables" where you paste the real value.

Your code asks for the value by name; the real value is entered separately, in a place that never enters your repository. Same key, two safe homes — never the code itself.

🌱 Learn the idea behind this The practice of keeping configuration and secrets out of code and in the environment is part of a widely-followed set of principles called the Twelve-Factor App. Its short, readable page on configuration is worth a look: 12factor.net/config.
Going live changes your exposure — read this honestly The moment you deploy, your app is reachable by anyone on the internet, including automated bots that scan constantly. A few things follow from that, and they are your responsibility, not the host's:
  • Every secret goes in the host's settings, never in code — a key pushed to a public repo can be found and abused within minutes.
  • If your app collects anyone's personal data, you are now responsible for handling it lawfully and carefully. That's a real obligation, not a checkbox.
  • Hosting being easy does not make your app secure. For anything that handles sensitive information, get a qualified review before you put it in front of real users.

    What you just learned

    • Deploy = make your app reachable on the internet for real users.
    • "The cloud" is just rented computers someone else keeps running for you.
    • Hosting (Netlify, Vercel) deploys automatically when you push to GitHub.
    • In production, secrets live in the host's settings as environment variables — never in code.
    • Going live increases your exposure and your responsibility; easy hosting is not the same as a secure app.
